# pair-decision-recorder

페어 프로그래밍 의사결정을 추적 가능한 형태로 기록한다.

## 역할
- 페어 논의 내용에서 결정, 근거, 대안, 미해결 이슈를 구조화한다.
- 테스트 전략과 연결되는 의사결정을 남긴다.

## 입력 계약
- 확인 가능한 대화/커밋/테스트 결과만 근거로 사용한다.
- 불확실한 내용은 `TODO(확인 필요)`로 남긴다.
- 개인식별정보나 비밀정보는 기록하지 않는다.

## 실행 방법
1. 세션 범위와 주제를 정한다.
2. 논의 포인트를 결정 단위로 분해한다.
3. 각 결정에 대해 선택안, 근거, 대안, 영향, 담당, 목표일을 기록한다.
4. 미해결 이슈를 별도로 등록하고 후속 액션을 지정한다.
5. `PAIR_DECISIONS.md`에 누적 기록한다.

## 파일 기록 정책
- 파일이 없으면 생성한다.
- 파일이 있으면 기존 내용을 유지하고 새 세션을 append 한다.
- 세션 헤더 형식: `## Session YYYY-MM-DD HH:mm`

## 출력 형식

```markdown
# PAIR_DECISIONS

## Session YYYY-MM-DD HH:mm

### 세션 정보
- 참여자: [Human], [Claude]
- 주제: [예: API 5개 E2E 전략]
- 관련 문서: [TEST_STRATEGY.md, AI_USAGE_LOG.md]

### 논의 요약
- [핵심 논의 1]
- [핵심 논의 2]

### Decision Log
| 결정 ID | 날짜 | 주제 | 선택한 결정 | 근거 | 고려한 대안 | 영향 범위 | 담당 | 목표일 | 상태(확정/보류) | 근거 링크 |
|---------|------|------|-------------|------|-------------|-----------|------|--------|-----------------|-----------|
| D-001 | YYYY-MM-DD | [주제] | [결정] | [근거] | [대안] | [영향] | [담당] | [일자] | [상태] | [근거] |

### Rejected Alternatives
| 대안 | 기각 이유 | 재검토 조건 |
|------|-----------|-------------|
| [대안] | [사유] | [조건] |

### Open Issues
| 이슈 ID | 내용 | 막히는 이유 | 필요한 정보 | 담당 | 목표일 | 상태 |
|---------|------|-------------|-------------|------|--------|------|
| I-001 | [내용] | [이유] | [정보] | [담당] | [일자] | [상태] |

### 다음 액션
- [ ] [액션 1] (담당: [이름], 기한: [YYYY-MM-DD])
- [ ] [액션 2] (담당: [이름], 기한: [YYYY-MM-DD])
```

## 품질 체크리스트
- 모든 결정에 근거가 있는가?
- 결정 ID와 이슈 ID가 중복 없이 관리되는가?
- 담당과 목표일이 없는 항목이 없는가?
- `TEST_STRATEGY.md`의 행위/데이터/검증 전략과 모순이 없는가?

## 사용 예시

```text
/pair-decision-recorder
```

```text
/pair-decision-recorder 테스트 전략 수립 세션
```
